# API Reference

The BudgetingApp backend exposes a REST API built with FastAPI. All routes are mounted under the `/api`
prefix (see `backend/app/main.py`). Interactive OpenAPI documentation is auto-generated by FastAPI and
served at `/docs` (Swagger UI) and `/redoc` (ReDoc) whenever the backend is running.

## Authentication

There is no authentication mechanism in the codebase. Every endpoint is unauthenticated and accepts
requests from any origin permitted by CORS. `backend/app/main.py` configures CORS via
`CORSMiddleware` with `allow_origins=["http://localhost:5173"]` (the Vite dev server) and
`allow_credentials=True`, `allow_methods=["*"]`, `allow_headers=["*"]`.

`backend/app/main.py`.

## Request / response formats

### `GET /api/health`

No parameters. Response:

```json
{ "status": "ok" }
```

### `POST /api/upload`

Multipart form upload with a single `file` field (the bank CSV export). The backend detects the bank
format automatically (`backend/app/parsers/detector.py`) ŌĆö supported formats are Revolut French
(`revolut_fr`), Revolut English (`revolut_en`), and Cr├®dit Agricole (`ca`). Merged multi-currency Revolut
exports are rejected.

Request (as sent by `frontend/src/api/client.ts`):

```bash
curl -X POST http://localhost:8000/api/upload \
  -F "file=@statement.csv"
```

Success response:

```json
{
  "format_detected": "revolut_en",
  "bank": "Revolut",
  "total_parsed": 42,
  "imported": 40,
  "duplicates_skipped": 2,
  "batch_id": 7
}
```

Each parsed transaction is categorized via `backend/app/services/categorizer.py` (OpenAI `gpt-4o-mini`,
batches of 30), then converted to the base currency (`ILS`, hardcoded in `upload.py`) via
`backend/app/services/currency.py`. Duplicate detection is based on an exact match of date, original
amount, original currency, bank, and description (see the `uq_transaction_dedup` constraint on the
`Transaction` model in `backend/app/models/transaction.py`).

Error responses for unsupported/invalid uploads are returned with HTTP status `200` and an `error` key
in the JSON body (there is no `HTTPException` usage in the upload router):

```json
{ "error": "Merged Revolut CSVs are not supported. Please upload per-currency account statements." }
```

```json
{ "error": "Unsupported format: <detected-format>" }
```

```json
{ "error": "No valid transactions found in the CSV" }
```

### `PATCH /api/transactions/{transaction_id}/category`

Request body:


Response on success: `{"ok": true}`. Response if the transaction does not exist: `{"error": "Transaction not found"}` (HTTP status `200`, not `404`).

Request body:


`display_order` is optional (defaults to `0`). Response: `{"id": <int>, "name": <string>}`.

Request body:

### `POST /api/budget/targets`

Request body:


## Error codes

There is no centralized error-handling middleware or custom exception handler in the codebase (no
`HTTPException` or `@app.exception_handler` usage was found in `backend/app`). Error behavior is:

- **Business-logic errors** (unsupported CSV format, transaction/category not found, etc.) are returned
  as HTTP `200 OK` with a JSON body of the form `{"error": "<message>"}`. Callers must check for the
  presence of an `error` key rather than relying on the HTTP status code.
- **Request validation errors** (e.g., a missing required field, wrong type in a path parameter) return
  FastAPI's default `422 Unprocessable Entity` with the standard FastAPI validation error body.
- **Unhandled exceptions** (e.g., a database error) fall through to FastAPI's default `500 Internal
  Server Error` handler.

| Status Code | Meaning |
|-------------|---------|
| `200` | Success, or a handled business-logic failure (check for an `error` key in the body) |
| `422` | Request failed FastAPI/Pydantic validation (malformed query/path parameters or request body) |
| `500` | Unhandled server-side exception |
